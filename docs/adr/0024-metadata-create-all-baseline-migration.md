# 0024 — Baseline migration builds the schema from the canonical MetaData

- Status: Accepted
- Date: 2026-06-22

## Context

The relational schema (fourteen tables) is defined once as a SQLAlchemy Core
`MetaData` in `adapters/postgres/schema.py` — the single source of truth the
application queries against. Alembic's first ("baseline") revision must create
that schema. The conventional approach transcribes each table into the revision
as `op.create_table(...)` calls, which duplicates every column, constraint, and
index and invites silent drift between the revision and the model.

## Decision

The baseline revision builds the schema by calling `metadata.create_all` (and
`drop_all` on downgrade) against the canonical `MetaData` it imports from
`schema.py`. Alembic's `env.py` sets `target_metadata` to that same `MetaData`.
An integration test asserts that every model table exists after `upgrade head`.

## Consequences

- The baseline and the model are identical by construction — there is no second
  transcription of the schema to keep in sync, and constraint names come from the
  `MetaData` naming convention, so the generated DDL is deterministic.
- `target_metadata` still points at the canonical `MetaData`, so subsequent
  incremental changes use ordinary `alembic revision --autogenerate` and compare
  cleanly against a database that was built from that same metadata.
- The baseline is a single atomic "create everything" step rather than a
  per-table narrative. That suits an initial schema; once published it is
  immutable like any accepted migration, and later structural changes ship as new
  autogenerated revisions.
